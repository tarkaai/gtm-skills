# Power User Program — Durable Intelligence

> **Stage:** Product → Referrals | **Motion:** Lead Capture Surface | **Channels:** Product, Email

## Outcomes

The advocacy program runs on autopilot. The AI optimization agent continuously monitors enrollment, activation, referral, and retention metrics. When any metric plateaus or declines, the agent diagnoses the cause, generates improvement hypotheses, runs A/B experiments, evaluates results, and auto-implements winners. The program sustains at least 50 active advocates producing 3+ actions per quarter each over 6 months. The system converges toward the local maximum of advocacy-driven growth for the current user base.

## Leading Indicators

- AI detects anomalies within 24 hours of metric deviation (daily health check classifies correctly)
- Hypotheses generated by the AI produce positive experiment results at least 40% of the time
- Weekly optimization briefs show cumulative positive AI lift trend
- Advocacy health monitor triggers fewer escalations over time (system stabilizing)
- Referral conversion rate sustains or improves quarter over quarter
- Convergence signal: successive experiments produce <2% improvement for 3 consecutive experiments

## Instructions

### 1. Deploy the autonomous optimization loop

Run the `autonomous-optimization` drill configured for the advocacy program:

**Phase 1 — Monitor (daily via n8n cron):**
Configure the monitoring loop to track advocacy-specific KPIs:
- Primary KPIs: active advocate count (90-day rolling), first-action activation rate (30-day), referral conversion rate (4-week rolling), advocacy actions per advocate per quarter, revenue attributed to referrals
- Use `posthog-anomaly-detection` to compare last 2 weeks against 4-week rolling average
- Classification: normal (within +/-10%), plateau (+/-2% for 3+ weeks), drop (>20% decline), spike (>50% increase)
- If normal: log to Attio, no action
- If anomaly detected: trigger Phase 2

**Phase 2 — Diagnose (triggered by anomaly):**
Gather advocacy program context from PostHog and Attio:
- Current tier distribution and per-tier engagement rates
- Enrollment funnel conversion rates by week
- Nudge sequence performance (open rates, click rates, action rates)
- Referral funnel conversion at each stage
- Recent experiment history and outcomes
Run `hypothesis-generation` with the anomaly data. Receive 3 ranked hypotheses. Examples:
- "Activation rate dropped because the testimonial ask is stale after 3 months — rotating to a new ask format (video instead of written) will restore activation by 12pp"
- "Referral conversion declined because the referee landing page load time increased from 1.2s to 3.8s after last deploy — fixing load time will restore conversion to baseline"
- "Enrollment rate plateaued because the eligible pool growth slowed — lowering the Insider threshold from 60 to 55 will expand the pipeline by 30% without degrading activation quality"
- "Advocate retention is declining because Ambassador benefits are not differentiated enough — adding a quarterly product strategy call with the CEO will improve 90-day retention by 15pp"
If top hypothesis risk = "high" (e.g., lowering threshold, changing tier structure), escalate to human review and stop.

**Phase 3 — Experiment (triggered by hypothesis acceptance):**
Design and launch the experiment:
- Use `posthog-experiments` to create a feature flag splitting traffic between control and variant
- Implement the variant (e.g., new ask format, adjusted threshold, new benefit)
- Minimum duration: 7 days or 50+ users per variant (advocacy experiments have smaller sample sizes than product-wide experiments)
- Log experiment in Attio: hypothesis, start date, expected duration, success criteria
- Maximum 1 active experiment at a time per `autonomous-optimization` guardrails

**Phase 4 — Evaluate (triggered by experiment completion):**
Pull results from PostHog. Run `experiment-evaluation`:
- **Adopt**: improvement is statistically significant (90% confidence for small-sample advocacy tests) and practically meaningful (>5% relative improvement). Roll out to 100%. Log the change.
- **Iterate**: result is directionally positive but not significant. Generate a refined hypothesis. Return to Phase 2.
- **Revert**: variant performed worse or no difference. Restore control. Log the failure. Return to Phase 1.
- **Extend**: insufficient sample size. Continue for another 7 days.
Store evaluation in Attio with decision, confidence level, and reasoning.

**Phase 5 — Report (weekly via n8n cron):**
Generate weekly optimization brief:
- Anomalies detected this week and classifications
- Hypotheses generated and their status
- Active experiment progress (sample size, preliminary direction)
- Completed experiments: adopted, reverted, or iterated
- Net metric change from all adopted changes this week
- Current distance from estimated local maximum (based on improvement trend)
- Recommended focus for next week
Post to Slack and store in Attio.

### 2. Deploy the advocacy health monitor

Run the `autonomous-optimization` drill for play-specific monitoring that complements the generic optimization loop:

- Configure daily health checks for all 8 advocacy metrics: recruitment rate, activation rate, Insider-to-Advocate conversion, referral yield, referral conversion, advocate retention, power user score stability, testimonial/content yield
- Implement diagnostic triggers for each declining metric:
  - Recruitment declining: check if eligible pool shrank (product engagement upstream) vs. enrollment mechanic broken (stale messaging)
  - Activation declining: check nudge sequence performance by stage (Day 7, 14, 21), compare by enrollment cohort
  - Referral yield declining: check if advocates are sharing links at all vs. sharing but not converting
  - Advocate retention declining: cross-reference with product usage to distinguish program churn from product churn
- Configure automated interventions:
  - Stale enrollment message: auto-rotate to next variant if click rate drops below 5%
  - Activation stall: extra personalized nudge for cohorts with <20% 14-day activation
  - Referral drought: "refer and earn" reminder to top 10 advocates if no referrals in 14 days
  - Lapsed advocate: re-engagement email with fresh ask based on recent product usage
- Build weekly health report: aggregate daily checks, trend analysis, intervention outcomes, notable achievements
- Set escalation rules: any metric critical 3+ days, advocate retention below 50% for 2 weeks, referral conversion below 10% for 4 weeks, 3+ interventions in one week with no improvement

### 3. Monitor for convergence

The optimization loop runs indefinitely. Watch for convergence signals:

- 3 consecutive experiments produce <2% improvement on the target metric
- All 8 health metrics stable in "healthy" range for 4+ consecutive weeks
- Weekly optimization briefs consistently report "no anomalies detected"

At convergence:
1. The advocacy program has reached its local maximum
2. Reduce monitoring frequency from daily to weekly
3. Reduce experiment cadence from continuous to monthly spot-checks
4. Report: "Power User Program is optimized at [current metrics]. Active advocates: [N]. Actions per quarter: [N]. Referral conversion: [N]%. Revenue attributed: $[N]/mo. Further gains require strategic changes (new advocacy channels, new reward types, product-level virality features) rather than tactical optimization."

### 4. Handle structural changes

If the optimization loop stalls (4 consecutive failed experiments) or health metrics enter critical despite optimization:

**Human action required:** The agent flags this for strategic review. Possible structural changes:
- Add new advocacy channels: public community, social media ambassador program, event speaking opportunities
- Redesign the reward structure: switch from credits to tangible rewards, add gamification (advocacy leaderboard with prizes)
- Integrate with product virality: built-in sharing features, collaborative workspaces that naturally invite new users, public galleries of user work
- Expand the program scope: partner advocacy (agencies/consultants who recommend the product), developer advocacy (API users building integrations)
- Reconsider the scoring model: new dimensions, different weights, additional data sources

After structural changes are implemented, reset the optimization baseline and restart the loop.
