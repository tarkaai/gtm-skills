# G2/Capterra Review Requests — Durable Intelligence

> **Stage:** Product → Referrals | **Motion:** Lead Capture Surface | **Channels:** Product, Email

## Outcomes

The review request machine runs on autopilot. The AI optimization agent continuously monitors the ask-to-review funnel, review velocity, platform balance, ask fatigue, and review sentiment. When any metric plateaus or declines, the agent diagnoses the cause, generates improvement hypotheses, runs A/B experiments via PostHog feature flags, evaluates results, and auto-implements winners. The system sustains at least 30 reviews per month with an average rating of 4.0+ over 6 months. The optimization loop converges toward the local maximum of review-driven social proof for the current user base and product.

## Leading Indicators

- AI detects funnel anomalies within 24 hours of metric deviation (daily health check classifies correctly)
- Hypotheses generated by the AI produce positive experiment results at least 40% of the time
- Weekly optimization briefs show cumulative positive AI lift trend
- Review request health monitor triggers fewer escalations over time (system stabilizing)
- Ask fatigue metrics remain stable or decline over 6 months (the agent manages cooldowns and rotation effectively)
- Review velocity sustains or improves quarter over quarter
- Convergence signal: successive experiments produce <2% improvement for 3 consecutive experiments

## Instructions

### 1. Deploy the autonomous optimization loop

Run the `autonomous-optimization` drill configured for the review request play:

**Phase 1 -- Monitor (daily via n8n cron):**
Configure the monitoring loop to track review-request-specific KPIs:
- Primary KPIs: reviews per month (rolling 4-week), average review rating (rolling 4-week), show-to-click rate (rolling 4-week), click-to-submit rate (rolling 4-week), ask fatigue rate (rolling 4-week)
- Secondary KPIs: reviews per platform per week, trigger type distribution, follow-up email conversion rate, candidate pipeline depth
- Use `posthog-anomaly-detection` to compare last 2 weeks against 4-week rolling average
- Classification: normal (within +/-10%), plateau (+/-2% for 3+ weeks), drop (>20% decline), spike (>50% increase)
- If normal: log to Attio, no action
- If anomaly detected: trigger Phase 2

**Phase 2 -- Diagnose (triggered by anomaly):**
Gather review request context from PostHog, Intercom, and Attio:
- Current prompt variants and their per-variant conversion rates
- Trigger type performance breakdown (CTR, completion rate, avg rating per trigger)
- Platform-specific conversion rates and review volume
- Ask fatigue trend (dismiss rate, never-ask rate, cooldown pool size)
- Review candidate pipeline depth and eligible user growth rate
- Recent experiment history and outcomes

Run `hypothesis-generation` with the anomaly data. Receive 3 ranked hypotheses. Examples:
- "Click-to-submit rate dropped because users are landing on G2's updated review form which added 3 new required fields -- switching the primary CTA to Capterra (simpler form) will restore completion rate by 8pp"
- "Show-to-click rate plateaued because the winning prompt copy has been running for 8 weeks and users have developed banner blindness -- rotating to a fresh copy variant will restore CTR by 5pp"
- "Ask fatigue rate spiked because a product update increased engagement scores for 200+ users who simultaneously became eligible -- implementing a staggered rollout (max 20 new asks per day) will prevent batch fatigue"
- "Review velocity dropped because the post-support trigger stopped firing after an Intercom workflow update -- checking and repairing the n8n trigger connection will restore 30% of review volume"
- "Average rating declined to 3.8 because the engagement threshold was lowered to 50 in last month's experiment -- reverting to 60 will restore rating to 4.2+ while reducing volume by only 10%"

If top hypothesis risk = "high" (e.g., changing the engagement threshold, modifying platform routing for all users, adjusting cooldown periods below 14 days), escalate to human review and stop.

**Phase 3 -- Experiment (triggered by hypothesis acceptance):**
Design and launch the experiment:
- Use `posthog-experiments` to create a feature flag splitting traffic between control and variant
- Implement the variant via Intercom (new message variant) or n8n (modified trigger logic)
- Minimum duration: 7 days or 50+ impressions per variant (review request experiments have moderate sample sizes)
- Log experiment in Attio: hypothesis, start date, expected duration, success criteria
- Maximum 1 active experiment at a time per `autonomous-optimization` guardrails

**Phase 4 -- Evaluate (triggered by experiment completion):**
Pull results from PostHog. Run `experiment-evaluation`:
- **Adopt**: improvement is statistically significant (90% confidence) and practically meaningful (>5% relative improvement on the target metric). Roll out to 100%. Log the change. Update the Intercom message variant or n8n trigger configuration.
- **Iterate**: result is directionally positive but not significant. Generate a refined hypothesis. Return to Phase 2.
- **Revert**: variant performed worse or no difference. Restore control. Log the failure. Return to Phase 1.
- **Extend**: insufficient sample size after minimum duration. Continue for another 7 days.
Store evaluation in Attio with decision, confidence level, and reasoning.

**Phase 5 -- Report (weekly via n8n cron):**
Generate weekly optimization brief:
- Anomalies detected this week and classifications
- Hypotheses generated and their status
- Active experiment progress (sample size, preliminary direction)
- Completed experiments: adopted, reverted, or iterated
- Net metric change from all adopted changes this week
- Current review velocity: {N} reviews/month, avg rating {N}, platform split {G2%}/{Capterra%}
- Ask fatigue trend: dismiss rate {N}%, never-ask rate {N}%
- Current distance from estimated local maximum (based on improvement trend)
- Recommended focus for next week
Post to Slack and store in Attio.

### 2. Deploy the review request health monitor

Run the `review-request-health-monitor` drill for play-specific monitoring that complements the generic optimization loop:

- Configure the PostHog "Review Request Health" dashboard with all 6 panels: ask-to-review funnel, timing effectiveness, platform split, engagement score distribution of reviewers, ask fatigue signals, review quality
- Implement daily health checks (08:00 UTC via n8n) covering: ask funnel health, timing window health, platform balance, ask fatigue, review sentiment
- Configure automated interventions:
  - Stale timing trigger: reduce frequency of underperforming trigger types, increase weight on best performers
  - Ask fatigue: extend cooldowns, rotate prompt copy, tighten eligibility threshold if "never ask" rate exceeds 10%
  - Platform imbalance: rotate primary platform when one receives <30% of reviews for 2 weeks
  - Negative review spike: pause all asks for 48 hours if 3+ negative reviews in 7 days, alert CS team, queue vendor response drafts
  - Review drought: diagnose whether asks are showing (prompt broken) or showing but not converting (copy/timing issue)
- Build weekly health report with funnel performance, timing effectiveness rankings, platform health, fatigue signals, and intervention log
- Maintain review candidate pipeline in Attio with pipeline depth tracking and eligibility management

### 3. Monitor for convergence

The optimization loop runs indefinitely. Watch for convergence signals:

- 3 consecutive experiments produce <2% improvement on the target metric
- All review request health metrics stable in "healthy" range for 4+ consecutive weeks
- Weekly optimization briefs consistently report "no anomalies detected"
- Ask fatigue metrics are stable and within acceptable range (<50% dismiss, <8% never-ask)

At convergence:
1. The review request play has reached its local maximum
2. Reduce monitoring frequency from daily to weekly
3. Reduce experiment cadence from continuous to monthly spot-checks
4. Report: "In-App Review Request is optimized at [current metrics]. Review velocity: [N]/month. Average rating: [N]. Platform split: G2 [N]% / Capterra [N]%. Ask fatigue: [N]% dismiss, [N]% never-ask. Candidate pipeline: [N] weeks. Further gains require strategic changes (new review platforms, new ask channels like email-only campaigns, or product-level virality features) rather than tactical optimization."

### 4. Handle structural changes

If the optimization loop stalls (4 consecutive failed experiments) or health metrics enter critical despite optimization:

**Human action required:** The agent flags this for strategic review. Possible structural changes:
- Add new review platforms: TrustRadius, Product Hunt, industry-specific directories
- Add new ask channels: standalone email campaigns (not just in-app), post-renewal asks, customer success check-in asks
- Incentive programs: G2 gift card campaigns (compliant with G2 guidelines -- incentivize the act of reviewing, not the rating)
- Video reviews: ask power users for video testimonials that G2 features prominently (higher weight in rankings)
- Review recycling: ask users who reviewed 12+ months ago to update their review (platforms weight recent reviews more heavily)
- Product-level changes: build a "review us" surface in the product settings or help center, not just timed prompts

After structural changes are implemented, reset the optimization baseline and restart the loop.
